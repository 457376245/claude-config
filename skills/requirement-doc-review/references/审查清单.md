# 审查清单

执行 Requirement Doc Review 时，用这份清单约束审查范围。目标是判断“当前代码是否满足需求实施追踪文档”，不是做泛泛的代码美化。

## 输入完整性

- 是否找到了唯一明确的需求实施追踪文档？
- 文档是否包含原始需求、范围边界、假设和验收标准？
- 文档是否包含最终一致性检查或阶段性交接说明？
- 文档是否记录实际验证结果和未验证事项？

## 需求达标

- 每条验收标准是否能在代码、测试、配置或文档中找到证据？
- 是否存在文档说已交付但代码没有实现的行为？
- 是否存在代码实现超出范围，且文档未说明的行为？
- 是否存在本次明确“不做”的内容被误实现或被实现依赖？

## 技术一致性

- 文档列出的文件、模块、接口、路由、表、配置、任务是否真实存在并被正确修改？
- 代码中实际修改的关键文件是否已记录在文档中？
- 接口、数据库、权限、状态流转、定时任务、MQ、缓存等影响是否与文档一致？
- 跨项目或外部依赖是否记录了真实契约和依赖顺序？

## 验证可靠性

- 已运行测试是否覆盖核心验收标准？
- 未运行测试是否有明确原因？
- 是否存在只能通过人工、环境或外部系统确认的事项？如果有，标记为 `无法确认`。
- 是否有必要补充单元测试、集成测试、接口测试或手工验证步骤？

## 报告要求

- 结论先行，阻塞问题优先。
- 每个问题尽量引用文件、行号、测试、命令结果或文档章节。
- 区分 `不通过`、`部分通过`、`无法确认`，不要把未知事项写成通过。
- 不直接修改业务代码，除非用户明确要求修复。
- 不静默修改需求实施追踪文档；需要同步时在报告里提出。
