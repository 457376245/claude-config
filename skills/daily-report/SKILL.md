# 日报助手

## 核心工作流

### 1. 接收用户输入

用户调用时需提供：
- **commit 信息**：commit hash、hash 范围（如 `abc123..def456`）、或分支名
- **git 仓库路径**（可选）：默认为当前工作目录

通过 `AskUserQuestion` 确认：
- 用户指定的 commit 范围
- 如未指定仓库路径，询问仓库位置

### 2. 校验 commit 去重

Commit 记录文件路径：`E:\ZHONGZHAI\docs\日报\{YYYY-MM}日报-commits.txt`

**文件格式：**
```
# {YYYY-MM} 已处理 commit 记录
2b099f1317b7f58a714cb3db8bb70f6b92d70158 feat: 实现用户登录前端表单
93f3fbf382c9f272655f77867409e9771213789f fix: 修复 token 存储问题
```

**校验流程：**
1. 读取 commit 记录文件（不存在则视为空）
2. 对用户指定的每个 commit，获取其完整 hash：`git -C <repo_path> rev-parse <commit>`
3. 检查完整 hash 是否出现在记录文件某行的行首（用 `grep -F "<full_hash>" <commits_file>`）
4. 若已存在 → 跳过该 commit，通知用户"该 commit 已填入过日报"
5. 若所有 commit 都已处理过 → 直接结束，无需后续操作
6. 仅对未处理的 commit 继续后续步骤

**记录时机：** 在日报内容成功填入文件后，执行以下命令获取格式化记录行，再追加到记录文件末尾：
```bash
git -C <repo_path> log --pretty=format:"%H %s" -1 <commit>
```
输出示例：`2b099f1317b7f58a714cb3db8bb70f6b92d70158 feat: 实现用户登录前端表单`

将该完整输出行追加到记录文件。若记录文件不存在则创建，首行写入注释 `# {YYYY-MM} 已处理 commit 记录`。

### 3. 检查/创建日报文件

日报路径：`E:\ZHONGZHAI\docs\日报\{YYYY-MM}日报.txt`（YYYY-MM 为当前年月）

**检查文件是否存在：**
```bash
ls "E:/ZHONGZHAI/docs/日报/{YYYY-MM}日报.txt"
```

**若不存在，创建并初始化：**

1. 读取 `references/holidays-{YYYY}.md` 获取当年节假日和调休工作日
2. 生成当月所有工作日（排除周末 + 排除法定假日 + 包含调休工作日）
3. 按如下格式初始化文件：

```
2026-04-01：
2026-04-02：
2026-04-03：
...（仅列出工作日）
```

**判断工作日的规则：**
- 默认周一至周五为工作日，周六周日为非工作日
- 若日期出现在节假日表的"放假日期"中 → 非工作日
- 若日期出现在节假日表的"调休工作日"中 → 工作日（即使是周末）

### 4. 分析 commit 并估算人天

对用户指定的 commit 执行：

```bash
git -C <repo_path> log --oneline <commit_range>
git -C <repo_path> diff <commit_range> --stat
git -C <repo_path> diff <commit_range>
```

**人天估算规则：**
- 查看 diff 涉及的文件数、修改行数、逻辑复杂度
- 参考标准：
  - 简单改动（配置、文案、小修复）：~0.5 天，向上取整为 1 天
  - 中等功能（一个完整模块/接口）：1-2 天
  - 复杂功能（跨模块、涉及多个文件的重构）：3-5 天
  - 大型特性：5+ 天
- 将总量拆分成每天可描述的合理工作粒度
- 通过 `AskUserQuestion` 向用户展示估算结果和拆分方案，用户确认或调整后再填入

### 5. 生成日报内容

将 commit 的工作拆解为每天的详细分点内容。

**每天的格式：**
```
2026-04-03：
  1. 实现用户登录前端表单，包括账号密码输入、验证码组件
  2. 编写登录接口对接代码，处理 token 存储
```

**拆分原则：**
- 每天 2-4 个要点，描述具体做了什么
- 子项使用 `1.  2.  3.` 编号格式，不使用 `-` 列表
- 内容要合理、自然，像真实的每日工作产出
- 前几天偏向基础搭建和核心实现，后几天偏向测试、优化、联调
- 避免一天内容过少（仅 1 项）或过多（超过 5 项）
- **填写范围不受当前日期限制**：工作量多可填到未来日期，工作量少可在任意已过日期结束

### 6. 填入日报文件

读取现有日报文件，找到最后一个已填写内容的日期，从下一个空日期开始填入。

**判断已填写：** 日期行后有缩进的内容行（以 `  1. `、`  2. ` 等编号开头的行）

**填入流程：**
1. 读取日报文件全部内容
2. 找到最后一个有内容的日期位置
3. 从下一个空白日期开始，按工作量顺序填入生成的日报内容（不限制必须填到当天，工作量决定填写的天数范围）
4. 写回文件
5. 将本次处理的 commit 追加到 `{YYYY-MM}日报-commits.txt`

### 7. 日报已满时的均匀重排

当所有工作日都已填写，没有空位时：

1. 收集日报中所有已填写的工作内容条目
2. 加入新 commit 生成的内容条目
3. 计算可用工作日总数
4. 将所有条目按时间逻辑均匀分配到各工作日
5. 每天分配 2-4 个要点，保持合理性
6. 通过 `AskUserQuestion` 展示重排方案让用户确认
7. 确认后覆盖写入日报文件

**重排原则：**
- 保持工作内容的逻辑顺序（基础搭建 → 核心开发 → 测试优化）
- 尽量均匀分配，避免某天过多或过少
- 如果内容总量超出工作日容量，提示用户需要精简

## 注意事项

- 所有文件操作前通过 `AskUserQuestion` 向用户确认
- 日报文件编码为 UTF-8
- 节假日参考文件位于 `references/holidays-{YYYY}.md`，如当年文件不存在需提醒用户创建
- 日报路径中的目录如不存在需先创建
- 不要修改已有日期的已填内容（除非执行重排）
